# 0. 이 노트의 큰 흐름

이 필기는 단순히 `CPU만 배우는 컴퓨터구조`가 아니라, 다음 흐름으로 묶어 보는 것이 공부에 훨씬 편합니다.

1. **컴퓨터 시스템 전체 그림**  
   하드웨어, 소프트웨어, 개발/운영, 품질, 시스템 구성
2. **CPU와 메모리, 저장장치, I/O**  
   컴퓨터가 실제로 어떻게 동작하는가
3. **프로그램 실행 구조**  
   컴파일, 로딩, 명령어, 자료형, 메모리 구조
4. **수와 데이터 표현**  
   진수, 보수, 부호, 비트, 실수 표현
5. **디지털 논리와 오류 검출**  
   불 대수, 논리게이트, 카르노 맵, 패리티, CRC

즉, 시험 대비 관점에서 보면 다음 한 줄로 요약할 수 있습니다.

```text
시스템 이해 -> 실행 원리 -> 데이터 표현 -> 논리 회로
```

# 1. OT 및 주변 개념 정리

## 1-1. 수업/필기에서 보이는 큰 방향

필기 첫 장은 순수한 컴퓨터구조 개념만이 아니라, 컴퓨터 시스템을 둘러싼 주변 개념까지 함께 적혀 있습니다.

- AI Studio, 노트북 LM, AI + GPT
- Agile
- DevOps
- CI / CD
- JIT, LEAN, Just In Time
- QC vs QA
- reactive vs proactive
- Meta Data
- 프로젝트, 시스템, 프로그램

이 부분은 시험 문제의 직접 정답보다도, **컴퓨터를 단순한 기계가 아니라 시스템과 개발 프로세스 전체로 보는 시각**을 잡는 데 의미가 있습니다.

## 1-2. Agile / DevOps / CI/CD

### Agile
- 변화에 빠르게 대응하려는 개발 방식
- 문서보다도 **반복적 개선**, **피드백**, **협업**을 중시

### DevOps
- 개발(Development)과 운영(Operations)을 분리하지 않고 연결해서 보는 관점
- 소프트웨어를 만드는 것에서 끝나는 것이 아니라, **배포 이후 운영과 개선까지 통합**

### CI / CD
- **CI(Continuous Integration)**: 코드를 자주 합치고 검증하는 흐름
- **CD(Continuous Delivery / Deployment)**: 검증된 결과물을 배포 가능한 상태로 유지하거나 실제 배포까지 자동화

정리하면:

```text
Agile = 개발 방식의 철학
DevOps = 개발과 운영을 잇는 문화/실천
CI/CD = 그 흐름을 자동화하는 기술적 체계
```

## 1-3. QC vs QA

### QC (Quality Control)
- 만들어진 결과물을 검사하고 문제를 찾는 쪽에 가까움
- 결과 중심
- 비교적 **reactive(사후 대응)** 성격

### QA (Quality Assurance)
- 문제가 생기지 않도록 과정 자체를 관리하는 쪽에 가까움
- 프로세스 중심
- 비교적 **proactive(사전 예방)** 성격

핵심 비교:

```text
QC = 결과 검사
QA = 과정 보증
```

## 1-4. 메타데이터와 시스템 관점

### Metadata
- 데이터를 설명하는 데이터
- 예: 파일명, 작성일, 형식, 크기, 작성자 등

### 시스템 / 프로그램 / 프로젝트
- **프로그램**: 특정 기능을 수행하는 코드 집합
- **시스템**: 프로그램 + 데이터 + 입출력 + 운영 환경까지 포함한 전체 구조
- **프로젝트**: 일정한 목표를 가지고 시스템/프로그램을 만드는 활동

# 2. HW + SW의 기본 구조

## 2-1. command vs instruction

필기상 `command`와 `instruction`이 비교되어 있습니다.
공부할 때는 아래처럼 잡는 것이 가장 이해하기 쉽습니다.

### Command
- 사용자가 내리는 명령, 또는 상위 수준의 요청
- 예: 실행하라, 저장하라, 삭제하라

### Instruction
- CPU가 이해하고 실행하는 기계 수준 명령
- 실제 하드웨어가 처리하는 단위

즉,

```text
사용자 관점의 명령 -> 내부적으로 instruction들의 실행으로 이어짐
```

## 2-2. task / job / work

필기에는 `Task vs Job - Work`가 적혀 있습니다.
엄밀한 정의는 문맥마다 조금 다르지만, 공부용으로는 아래처럼 잡으면 됩니다.

- **Work**: 가장 넓은 의미의 일
- **Job**: 하나의 업무 단위, 처리 대상
- **Task**: job을 이루는 더 작은 실행 단위

즉,

```text
Work > Job > Task
```

라고 이해하면 구조를 잡기 쉽습니다.

## 2-3. 비트, 바이트, 단위

### 비트(bit)
- 정보의 최소 단위
- 0 또는 1

### 바이트(byte)
- 보통 8비트

### 기본 식

```text
n비트 -> 2^n개의 서로 다른 상태 표현 가능
8비트 -> 2^8 = 256가지
1KB = 2^10 byte
```

필기에는 GHz, 10^9, 2진 표현 등도 함께 적혀 있는데, 핵심은 다음입니다.

- 클록 주파수는 CPU 동작 속도와 관련됨
- 비트 수가 커질수록 표현 가능한 경우의 수가 증가함

## 2-4. 변수와 자료 구조의 아주 기초적인 구분

필기에는 변수의 범위, 생명주기, 자료 유형이 함께 적혀 있습니다.

### 변수의 범위(scope)
- **지역 변수(local)**: 특정 블록/함수 안에서만 사용
- **전역 변수(global)**: 더 넓은 범위에서 사용

### 변수의 생명주기
- **정적(static)**
- **동적(dynamic)**

### 자료의 형태
- 스칼라(하나의 값)
- 객체(object)
- 배열/리스트/연결 리스트 등

## 2-5. Java Collection 기초

필기상 컬렉션 관련 메모가 있습니다.

### Set
- 중복 허용 X
- 순서가 중요하지 않은 경우가 많음

### List
- 순서 있음
- 인덱스로 접근 가능

### Map
- key - value 구조

짧게 비교하면:

```text
Set = 중복 없는 집합
List = 순서 있는 목록
Map = 키-값 사전
```

## 2-6. XML vs JSON

필기에는 XML과 JSON 비교가 있습니다.

### XML
- 태그 중심 구조
- 구조가 명확하지만 비교적 장황함

### JSON
- 키-값 중심 구조
- 가볍고 사람이 읽기 쉬움
- 웹/REST API에서 매우 자주 사용

간단 비교:

```text
XML = 설명적이고 무거운 편
JSON = 간결하고 현대적 웹 환경에 친화적
```

## 2-7. 캐시 메모리

### 왜 필요한가?
- CPU와 메인 메모리 사이에는 속도 차이가 큼
- 이 차이를 줄이기 위해 중간에 빠른 저장공간이 필요함

### 캐시 메모리의 역할
- 자주 쓰는 데이터를 더 빠르게 접근하게 함
- 전체 시스템 성능 향상에 도움

핵심:

```text
캐시 메모리 = CPU와 주기억장치 사이의 속도 격차 완화 장치
```

# 3. 컴퓨터 하드웨어 구성

## 3-1. 컴퓨터 HW 구성 5단계 메모

필기에는 다음 요소들이 보입니다.

- CPU(제어/연산 장치)
- 메인 메모리 RAM
- 보조기억장치 HDD
- 입/출력 장치 I/O
- CPU 구성요소: ALU, 제어장치, 레지스터 세트

## 3-2. CPU의 내부 구성

### ALU (Arithmetic Logic Unit)
- 산술 연산, 논리 연산 수행

### Control Unit
- 명령어 해석, 각 장치 제어

### Register
- CPU 내부의 매우 빠른 저장 공간
- 즉시 필요한 값이나 주소를 잠시 저장

## 3-3. RAM vs ROM

### RAM (Random Access Memory)
- 읽기/쓰기 가능
- 실행 중인 프로그램과 데이터를 저장
- 휘발성 메모리

### ROM (Read Only Memory)
- 주로 읽기 중심
- 비휘발성 메모리
- 부팅 관련 정보 저장에 사용

### ROM 계열 메모
필기에 적힌 종류:
- ROM
- PROM
- EPROM
- EEPROM

큰 흐름은 다음과 같습니다.

```text
ROM -> 한 번 쓰고 주로 읽기
PROM -> 프로그램 가능
EPROM -> 지우고 다시 쓸 수 있음(자외선 방식 등)
EEPROM -> 전기적으로 지우고 다시 쓸 수 있음
```

## 3-4. 주기억장치 vs 보조기억장치

### 주기억장치
- CPU가 직접 빠르게 접근
- 속도 빠름
- 용량은 상대적으로 작음
- 보통 휘발성

### 보조기억장치
- 저장 중심
- 속도 느림
- 용량 큼
- 보통 비휘발성

핵심 trade-off:

```text
속도 ↑ -> 가격 ↑ -> 용량 ↓
```

## 3-5. 부팅(booting)

필기에는 부트 스트랩, 부팅, 부트 섹터 관련 메모가 있습니다.

### 부팅의 의미
- 컴퓨터 전원을 켠 뒤 운영체제를 메모리에 올려 실행 준비를 하는 과정

### 부트스트랩
- 운영체제를 메모리로 적재하도록 돕는 최소한의 시작 코드

즉,

```text
부팅 = 저장장치에 있던 OS를 메모리에 올려 실행 가능한 상태로 만드는 과정
```

# 4. 프로그램 실행과 시스템 버스

## 4-1. 시스템 버스 3종류

필기상 정리:

1. 제어 신호
2. 데이터
3. 주소

일반적으로는 다음과 같이 설명합니다.

### 주소 버스(address bus)
- 어디로 갈지 지정

### 데이터 버스(data bus)
- 실제 데이터 전달

### 제어 버스(control bus)
- 읽기/쓰기, 동기화 등 제어 신호 전달

## 4-2. CPU 관련 레지스터 메모

필기에는 다음이 보입니다.

- MAR (Memory Address Register)
- MBR (Memory Buffer Register)

### MAR
- 접근할 메모리 주소를 저장

### MBR
- 메모리와 주고받는 실제 데이터 임시 저장

짧게 정리하면:

```text
MAR = 주소 저장
MBR = 데이터 저장
```

## 4-3. 명령어 처리 단계

필기에는 기계 사이클이 다음 순서로 적혀 있습니다.

1. 인출(fetch)
2. 해석(decode)
3. 실행(execute)
4. 저장(store)

이건 시험에서 매우 중요한 흐름입니다.

### 1) Fetch
- 메모리에서 명령어를 가져옴

### 2) Decode
- 명령어 뜻을 해석함

### 3) Execute
- 실제 연산/제어 수행

### 4) Store
- 결과를 저장함

## 4-4. opcode / operand / mnemonic

### Opcode
- 어떤 연산을 할지 나타내는 부분

### Operand
- 그 연산의 대상

### Mnemonic
- 기계어를 사람이 읽기 쉽게 기호화한 표현
- 어셈블리 언어에서 많이 보임

예를 들면:

```text
ADD R1, R2
- ADD = opcode
- R1, R2 = operand
```

## 4-5. 어셈블리 / 링커 / 로더

필기 메모를 공부용으로 정리하면 다음과 같습니다.

### 어셈블리어
- 기계어에 가까운 저수준 언어
- mnemonic을 사용

### 링커(linker)
- 여러 목적 코드/라이브러리를 연결해서 실행 파일을 만드는 단계

### 로더(loader)
- 실행 파일을 메모리에 적재하는 단계

핵심 흐름:

```text
소스코드 -> 컴파일/어셈블 -> 링킹 -> 로딩 -> 실행
```

# 5. 메모리 구조와 자료형

## 5-1. Stack vs Heap

필기에는 stack, heap 메모가 적혀 있습니다.

### Stack
- 함수 호출, 지역 변수, 복귀 주소 등과 관련
- LIFO 구조
- `RET -> 복귀 주소` 메모가 있음

### Heap
- 동적 메모리 할당 영역
- 프로그램 실행 중 필요한 만큼 할당/해제

짧게:

```text
Stack = 호출 흐름 관리
Heap = 동적 객체 저장
```

## 5-2. primitive type vs reference type

### primitive type
- 값 자체를 저장
- 크기가 비교적 고정적
- 예: int, char, boolean 등

### reference type
- 실제 데이터의 위치(주소 개념)를 저장
- 배열, 객체, 리스트 등과 연결됨

핵심:

```text
primitive = 값 저장
reference = 주소/참조 저장
```

## 5-3. 순차 리스트 vs 연결 리스트

필기에는 배열형 구조와 연결 리스트 비교가 있습니다.

### 순차 리스트(배열 기반)
- 연속된 메모리 공간 사용
- 조회 빠름
- 중간 삽입/삭제는 불리할 수 있음

### 연결 리스트
- 노드가 포인터로 연결됨
- 삽입/삭제 유리할 수 있음
- 조회는 느릴 수 있음

비교:

```text
배열 = 조회 강함
연결 리스트 = 삽입/삭제 강함
```

## 5-4. Procedure vs Function

필기에 `procedure vs function` 메모가 있습니다.

- **Procedure**: 처리 절차 중심
- **Function**: 보통 값을 반환하는 계산 중심

프로그래밍 언어마다 경계는 조금 다르지만, 공부할 때는 이렇게 나누면 충분합니다.

# 6. 컴파일, 인터프리트, JIT

## 6-1. compile vs interpret

### Compile
- 실행 전에 미리 번역
- 대표 예시: C 계열

### Interpret
- 실행 시점에 해석하면서 실행
- 대표 예시로 스크립트 언어가 자주 언급됨

## 6-2. JIT (Just In Time)

필기에는 `JIT`, `Just In Time`이 등장합니다.

### JIT
- 완전히 미리 번역하는 것도 아니고, 완전히 한 줄씩 즉석 해석하는 것도 아닌 중간 성격
- 실행 시점에 필요한 부분을 기계어로 번역해 성능을 높이려는 방식

짧게 비교:

```text
컴파일 = 미리 번역
인터프리트 = 실행하면서 해석
JIT = 실행 도중 필요한 부분을 번역해 최적화
```

## 6-3. compile vs build

필기 메모상 구분:

### Compile
- 소스코드를 기계가 이해할 수 있는 형태로 번역하는 단계

### Build
- 컴파일뿐 아니라 링킹, 패키징 등 실행 가능한 결과물을 만드는 전체 과정

# 10. 수의 표현과 진법

이 부분은 시험에 정말 자주 나오는 핵심 영역입니다.

## 10-1. 진수 변환의 핵심 원리

### 10진수 -> 2진수
- 2로 계속 나누고 나머지를 아래에서 위로 읽음

### 2진수 -> 10진수
- 각 자리값에 `2의 거듭제곱`을 곱해 더함

예:

```text
1011(2) = 1*2^3 + 0*2^2 + 1*2^1 + 1*2^0 = 11(10)
```

## 10-2. 2진수 / 8진수 / 16진수 관계

### 8진수
- 2진수 3비트씩 묶음

### 16진수
- 2진수 4비트씩 묶음

예:

```text
1011 1011(2)
= BB(16)
```

## 10-3. RGB와 비트 수

필기에는 `RR GG BB`, `24비트`, `alpha 포함시 32비트` 메모가 있습니다.

### RGB
- 빨강(R), 초록(G), 파랑(B)
- 각 채널 8비트라면 총 24비트

### Alpha
- 투명도 채널
- 포함하면 보통 32비트

## 10-4. ASCII와 UTF

### ASCII
- 기본 문자 인코딩
- 보통 7비트 기반으로 설명
- 영문, 숫자, 기본 기호 표현

### UTF / 유니코드
- 훨씬 많은 문자를 표현하기 위한 체계
- 다국어 처리에 적합

짧게:

```text
ASCII = 제한된 기본 문자 집합
UTF = 다양한 언어를 위한 확장 문자 체계
```

# 11. 보수, 부호, 오버플로우

## 11-1. 1의 보수와 2의 보수

### 1의 보수

### 2의 보수
- 1의 보수 + 1

예:

```text
1010
1의 보수 -> 0101
2의 보수 -> 0110
```

```text
8비트 -> 2^8 = 256
signed 범위 -> -128 ~ 127
```

## 11-4. overflow / underflow

### Overflow
- 표현 가능한 범위를 넘어서는 경우

### Underflow
- 매우 작은 값 쪽에서 표현 한계를 벗어나는 경우

공부 포인트:
- 정수 범위 초과
- 부동소수점에서 너무 크거나 너무 작은 값

# 12. 부동소수점과 실수 표현

## 12-1. 고정소수점 vs 부동소수점

### 고정소수점
- 소수점 위치가 고정
- 구조가 단순하지만 범위/정밀도에 한계

### 부동소수점
- 소수점 위치를 움직일 수 있음
- 넓은 범위의 수 표현 가능

## 12-2. float vs double

### float
- 단정도
- 메모리를 덜 사용하지만 정밀도가 더 낮음

### double
- 배정도
- 메모리 사용은 더 크지만 정밀도가 더 높음

## 12-3. Bias(지수 편향)

필기에는 `Bias(127)` 메모가 있습니다.

### 의미
- 지수를 음수/양수 모두 다루기 위해 저장 시 일정 값을 더해 표현하는 방식

핵심 식:

```text
실제 지수 = 저장된 지수 - Bias
```

# 13. BCD 코드와 Gray 코드

## 13-1. BCD 코드

필기 핵심:
- 0 ~ 9의 10진 숫자를 4비트로 표현
- 16진수처럼 A~F를 쓰지 않음

예:

```text
9 -> 1001
```

### BCD 덧셈 보정
BCD에서 한 자리 결과가 9를 넘으면 `0110(=6)`을 더해 보정합니다.

예시 개념:

```text
8 + 7 = 15
BCD 한 자리 계산에서 9 초과 -> 6을 더해 보정
```

## 13-2. Gray 코드

### 핵심 특징
- 인접한 값끼리 **1비트만 변화**
- 센서, 위치 검출, 오류 감소 맥락에서 자주 언급

예시 순서:

```text
00 -> 01 -> 11 -> 10
```

### 장점
- 값이 바뀌는 순간 여러 비트가 동시에 변하면서 생길 수 있는 혼선을 줄임

# 14. 오류 검출: 패리티, CRC, ARQ, FCS

이 부분은 필기 후반부에 섞여 있으므로 따로 정리해 두는 것이 좋습니다.

## 14-1. detection vs recognition

필기에는 detection / recognition 비교가 있습니다.

- **Detection**: 있는지 감지
- **Recognition**: 무엇인지 인식

즉,

```text
Detection = 존재 확인
Recognition = 의미 판별
```

## 14-2. 패리티 비트

### 목적
- 오류를 **검출**하기 위한 간단한 방법

### 종류
- 짝수 패리티(even parity)
- 홀수 패리티(odd parity)

### 한계
- 오류를 찾아내는 데는 도움이 되지만, 일반적으로 스스로 정확히 고치지는 못함

## 14-3. CRC

### CRC (Cyclic Redundancy Check)
- 네트워크/저장장치 등에서 널리 쓰이는 오류 검출 방식
- 패리티보다 더 강력한 오류 검출에 사용

## 14-4. FCS

- 프레임의 오류 검사를 위한 체크 값으로 이해하면 됨
- CRC와 함께 자주 묶여 등장

## 14-5. ARQ

### ARQ (Automatic Repeat reQuest)
- 오류가 나면 재전송을 요청하는 방식

짧게:

```text
패리티/CRC/FCS = 오류를 찾아냄
ARQ = 다시 보내 달라고 요청함
```

# 17. 카르노 맵(K-map)

필기 마지막 부분의 핵심입니다.

## 17-1. 카르노 맵의 목적

- 불 대수식을 시각적으로 단순화하기 위한 도구
- 회로를 줄이고, 게이트 수를 줄이는 데 유리

## 17-2. 왜 Gray code 순서를 쓰는가?

필기에는 `00, 01, 11, 10` 순서가 적혀 있습니다.

이유는 인접 칸끼리 **1비트만 달라지게** 하기 위해서입니다.
그래야 묶을 때 의미가 자연스럽습니다.

## 17-3. 기본 절차

### 1단계
- 진리표에서 1인 칸을 해당 위치에 표시

### 2단계
- 1, 2, 4, 8개처럼 **2의 거듭제곱 개수**로 묶음

### 3단계
- 가능한 한 크게 묶음

### 4단계
- 묶인 칸들에서 변하지 않는 변수만 남겨 식을 만듦

## 17-4. 중요한 규칙

- 가장자리끼리도 인접으로 볼 수 있음
- 겹쳐 묶는 것도 가능
- 대각선은 인접 아님
- XOR/XNOR 형태는 일반 카르노 맵에서 바로 깔끔하게 안 보일 수 있음

핵심 기억 문장:

```text
카르노 맵은 1을 넣고, 크게 묶고, 안 변하는 변수만 남기는 도구다.
```

# 18. 시험 직전 압축 암기 포인트

## 시스템/하드웨어
- CPU 구성: ALU + 제어장치 + 레지스터
- 주기억장치 = RAM, 보조기억장치 = HDD/SSD 등
- 캐시는 CPU와 메모리 속도 차이를 줄임
- 버스 3종: 주소 / 데이터 / 제어
- MAR = 주소, MBR = 데이터

## 실행 과정
- fetch -> decode -> execute -> store
- opcode = 연산 종류
- operand = 연산 대상
- linker = 연결, loader = 적재
- compile은 번역, build는 더 큰 전체 과정

## 자료형/메모리
- stack = 호출/복귀/지역 변수
- heap = 동적 할당
- primitive = 값 저장
- reference = 참조 저장
- 배열은 조회 강점, 연결 리스트는 삽입/삭제 강점

## 수와 표현
- n비트 -> 2^n 상태
- 8비트 = 256가지
- 8진수는 3비트씩, 16진수는 4비트씩 묶음
- 2의 보수 = 1의 보수 + 1
- 8비트 signed 범위 = -128 ~ 127
- float < double (정밀도 기준)
- 실제 지수 = 저장 지수 - Bias

## 디지털 논리
- BCD는 0~9만 사용
- Gray 코드는 인접 값이 1비트만 다름
- parity는 오류 검출용
- CRC는 더 강한 오류 검출
- K-map은 Gray code 순서 사용
- 분배법칙, 흡수법칙은 꼭 암기
