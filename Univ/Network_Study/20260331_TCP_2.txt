RWND : Receive Window

Receiver는 Sender에게 상태를 계속 전달해주고, Sender는 그에 기반에 맞춰 데이터를 전달해줌.

문제가 발생하지 않도록 하게끔 설계되어 있음
리시버가 센더를 잘 제어할 수 있다.

TCP 세그먼트 페이로드 중

rwnd : 아직 읽지 않은 데이터, 남은 버퍼 공간

TCP Connection Management

TCP가 데이터를 주고 받기 전에 반드시 수행해야 하는 절차.
1. 핸드쉐이크(UDP처럼 바로 데이터 보내는 게 X) 확인 절차(확인 정보 교환)
- 양쪽 형태도 확인해야 함. 정상적인 응답 상태인지도 확인해야 함.(state)
- 논리적 연결, 연결 상태 유지 프로토콜

* 2-Way handshake를 사용하지 않는 이유
- 위험성, 네트워크가 이상적으로 동작하지 않음. 연결의 신뢰성 보장 어려움
- 상태 확인 X, 시간 개념 등을 반영하지 못함.
- 양쪽이 서로의 상태를 확인할 수 없음 -> 위험, 심각한 오류 유발 가능

! half open 커넥션의 문제 !
- 나중에 서버에 도달하는 경우, 커넥션이 이미 종료 상태에 있으면, 시스템 자원 낭비, 잘못된 연결 유지
- "반쪽만 열린 연결"
- 데이터 duplicate 발생 가능

-> 3-Way handshake가 필요한 이유
SYN Packet 전송(req) -> SYN Packet, ACK 넘버 전송(res) 
-> Syn Packet + ACK 넘버 확인에 대한 ACK 전송(req)

메세지 3번이 오가는 과정이 아닌, 양방향으로 각각의 초기 상태를 확인하는 과정

요청-응답-확인

TCP closing(종료)
- 양방향으로 이어짐, 각 방향이 독립적으로 종료됨
FIN (req) -> FIN, ACK (res) -> 종료!

## TCP Congestion
느리다 X 네트워크가 감당할 수 있는 수준을 넘어섬
-> 라우터 큐 길어짐, 계속 쌓임 ->
1. 지연 증가(queuing dalay?)

수신 송신의 문제 (Flow Control) <-> 네트워크 자체가 감당 못하는 것(TCP Congestion)

- 네트워크 Congestion 시나리오 1
하나의 라우터를 통해 공유 (R)
-> sender들이 보내는 속도가 증가 -> 링크 용량이 초과되면 버퍼 용량이 쌓임
- 네트워크 용량 포화 상태 -> sender 계속 보냄 -> 지연 무수히 증가(성능 저하)

- 네트워크 Congestion 시나리오 2
데이터 드랍 시 데이터 재전송 -> 같은 정보를 여러번 전달 + 다른 데이터들 ==> 더 부하가 갈 수 밖에 없음(악순환)


**3-19페이지**
버퍼의 이상적인 시나리오 -> 현실에서는 불가능(센더는 버퍼의 상태를 모름) 
-> **혼잡을 추정해야 하는 수밖에 없음**
