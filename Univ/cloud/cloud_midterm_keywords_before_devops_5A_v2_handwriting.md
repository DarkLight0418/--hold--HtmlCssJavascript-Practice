# 클라우드컴퓨팅2 중간 정리 키워드
## 범위: `5A. 데브옵스 개요` 이전까지

> 목적: A4 1장(앞뒤 2페이지) 정도로 빠르게 훑는 **핵심 키워드 압축본**
> 방식: **교안 핵심 + 손필기 보강 포인트 분리형**
> 사용법: 먼저 위의 정식 개념을 보고, 바로 아래의 손필기 보강 문장으로 암기 연결

## 0. 먼저 잡을 큰 흐름

- **클라우드 컴퓨팅** = 인터넷으로 컴퓨팅 자원 제공
- **클라우드 네이티브** = 클라우드의 장점을 극대화하도록 앱을 설계/개발/운영
- **핵심 축** = 컨테이너 + 마이크로서비스 + 오케스트레이션 + API + 자동화
- **운영 관점 핵심** = 확장성, 복원력, 이식성, 자동화, 관찰성
- **중요 사고 전환**
  - 서버를 직접 만지는 방식 -> 선언형/자동화 방식
  - 큰 하나의 앱 -> 작고 독립적인 서비스
  - 상태를 내부에 둠 -> 상태를 외부로 분리
  - 수동 배포/수동 운영 -> 자동 배포/자동 복구

## 1. 클라우드 컴퓨팅 vs 클라우드 네이티브

### 클라우드 컴퓨팅
- 인터넷 기반 리소스 제공 서비스
- 자원: 서버, 스토리지, 네트워크, DB
- **사용량 기반 비용 지불**
- 특징
  - 확장성
  - 온디맨드 프로비저닝
  - 비용 효율성
- 서비스 모델
  - **IaaS**: 인프라 제공
  - **PaaS**: 개발/배포 플랫폼 제공
  - **SaaS**: 완성 소프트웨어 제공
- 유형
  - 퍼블릭 / 프라이빗 / 하이브리드

### 클라우드 네이티브
- 클라우드에 **최적화된 앱 설계/개발/배포/운영 방식**
- 클라우드 장점 활용
  - 동적 확장성
  - 분산 처리
  - 복원력
- 구성 키워드
  - 컨테이너화
  - 마이크로서비스
  - 동적 오케스트레이션
  - API-First
  - CI/CD
  - 불변 인프라
- 특징
  - 민첩성
  - 확장성
  - 회복성
  - 이식성
  - 자동화

### 비교 한 줄
- **클라우드 컴퓨팅** = 클라우드 자원을 쓰는 것
- **클라우드 네이티브** = 클라우드에 맞게 앱 자체를 설계하는 것

## 2. 분산 시스템 / 12요소 앱 / SLA

### 분산 시스템
- 여러 컴퓨터가 네트워크로 연결된 전체 시스템
- 기대 효과
  - 안전성
  - 신뢰성
  - 고가용성
  - 경제성
- 흔한 착각
  - 네트워크는 항상 안정적이다 X
  - 지연은 무시 가능하다 X
  - 대역폭은 무한하다 X
  - 네트워크는 안전하다 X
  - 토폴로지는 고정이다 X
  - 관리자는 항상 하나다 X
  - 전송 비용은 공짜다 X

### CAP 이론
- **C** 일관성
- **A** 가용성
- **P** 네트워크 파티션 허용
- 파티션은 현실적으로 발생 가능
- SQL 쪽은 일관성 강조, NoSQL 쪽은 가용성 강조 흐름으로 자주 설명

### 12요소 앱 핵심 키워드
- 코드베이스 1개, 다양한 배포
- 종속성 명시적 선언
- 설정은 코드 밖 환경으로 분리
- backend service는 연결된 자원처럼 취급
- **build / release / run 분리**
- **stateless process**
- 서비스별 데이터 격리
- 동시성: 수평 확장
- 폐기 가능성: 빠른 시작/종료
- dev-prod 일치
- 로그 = 이벤트 스트림
- 관리 작업 = 일회성 프로세스

### SLA / SLO
- **SLA** = 서비스 수준 협약, 가용성 수치
- 여러 서비스 조합 시 **전체 가용성은 더 낮아짐**
- 예: 99.95% x 99.90% -> 99.85%
- **SRE** = 신뢰성/가용성 향상 관점

## 3. 쿠버네티스 핵심

### 왜 필요한가
- 도커만으로 실행은 가능
- 하지만 서비스가 커지면 운영 문제가 발생
  - 죽은 컨테이너 자동 복구 필요
  - 트래픽 증가 시 자동 확장 필요
  - 여러 서버에 배치 관리 필요
  - 무중단 배포 필요

### 정의
- **컨테이너 오케스트레이션 플랫폼**
- 자동 배포 / 확장 / 관리

### 핵심 구조
- **클러스터** = 여러 서버 묶음
- **컨트롤 플레인** = 관리/제어
- **워커 노드** = 실제 실행

### 컨트롤 플레인 구성 요소
- **API Server**: 프론트엔드
- **etcd**: 클러스터 데이터 저장소
- **Scheduler**: 파드 배치 노드 결정
- **Controller Manager**: 상태 감시, 원하는 상태 유지

### 워커 노드 구성 요소
- **kubelet**: 파드 실행/상태 관리
- **kube-proxy**: 네트워크 규칙/포워딩
- **container runtime**: 컨테이너 실행

### 핵심 오브젝트
- **Pod**: 최소 실행 단위
- **Service**: 고정 접근 주소, 로드밸런싱
- **ReplicaSet**: 파드 복제본 수 유지
- **Deployment**: 배포/업데이트/롤백
- **StatefulSet**: 순서/고유성/지속 스토리지 필요할 때
- **DaemonSet**: 모든 노드에 1개씩 실행
- **Job / CronJob**: 일회성 / 스케줄 작업
- **Volume**: 데이터 영속성
- **ConfigMap / Secret**: 설정 / 민감정보 분리
- **Ingress**: 외부 HTTP/HTTPS 라우팅

### 동작 원리
- 개발자가 **desired state 선언**
- 시스템이 현재 상태를 계속 감시
- 다르면 자동으로 맞춤
- 핵심 표현: **선언형 시스템**, **자동 복구**, **자동 확장**

### 장단점
- 장점: self-healing, auto-scaling, rolling update, 인프라 추상화
- 단점: 러닝 커브 큼, 설정 복잡, 소규모 서비스엔 과할 수 있음

### 한 줄 비유
- **도커 = 포장**
- **PM2 = 실행**
- **쿠버네티스 = 운영**

## 4. 컨테이너 / 서버리스 / 함수 / 마이크로서비스

### 컨테이너
- 앱 + 라이브러리 + 설정 + 실행환경을 묶은 배포 단위
- OS 커널 공유, 앱 간 격리
- 장점
  - 빠른 시작
  - 이식성
  - 높은 응집도
  - scale-out 적합
- 관련 키워드
  - 이미지
  - COW 파일 시스템
  - OCI
  - containerd
  - CRI

### 서버리스
- 인프라 관리는 클라우드 제공자가 담당
- 개발자는 코드 실행에 집중
- 사용한 만큼 비용 지불
- 형태
  - **FaaS**
  - **CaaS**

### 함수(FaaS)
- 짧은 수명
- 단순, 독립, 무상태 작업에 적합
- 이벤트 기반 처리
- 제약
  - 실행 시간 제한
  - 기능 하나에 집중
  - 의존성/환경 제약
  - 벤더 락인 위험
  - 온프레미스/자체 클러스터 실행 부적합한 경우 있음

### 마이크로서비스
- 작고 독립적
- 느슨한 결합, 높은 응집도
- 비즈니스 기능 단위 분리
- 팀 단위 소유/배포/운영
- 장점
  - 민첩성
  - 장애 격리
  - 독립 배포
  - 확장성
  - 관찰성 향상
- 과제
  - 분산 시스템 복잡성
  - 데이터 일관성
  - 네트워크 비용
  - 버전 호환성
  - 모니터링/로깅
  - 의존성 관리

### 함수 vs 서비스
- **함수**: 짧고 단순한 작업
- **서비스**: 오래 실행, 상태/하드웨어/유연성 필요

## 5. 클라우드 네이티브 앱 설계

### 설계 시 고려 요소
- 운영 효율성
- 보안
- 안정성
- 확장성
- 비용

### 운영 효율성
- 모든 것 자동화
- 모든 것 모니터링
- 모든 것 문서화
- 점진적 변화
- 장애 대비 테스트

### 보안(심층 방어)
- 소스 코드 저장소 보안
- 컨테이너 이미지 최소화
- 컨테이너 저장소 접근 제어
- 인증된 이미지 사용
- 클러스터 접근 보호
- RBAC 활성화
- 서비스 간 통신 보호
- 데이터 보호

### 안정성
- **reliability**: 장애가 있어도 동작 신뢰
- **availability**: 일정 시간 동안 사용 가능
- 수평 확장과 다중 인스턴스 중요

### 전통 구조 vs 클라우드 네이티브
- 전통 구조: 상태를 앱 인스턴스 내부에 두기 쉬움
- 클라우드 네이티브: **상태를 외부로 분리**
- 장애를 없애려 하기보다 **장애를 가정하고 설계**

### 상호작용 방식
- **오케스트레이션**: 중앙에서 흐름 제어
- **코레오그래피**: 이벤트/메시지 기반 느슨한 협력

### API 설계와 버저닝
- 전략
  - 단일 버전 강제
  - 포인트 투 포인트 다버전 유지
  - **하위 호환 유지가 가장 효율적**
- 버전 방식
  - 글로벌 버저닝
  - 리소스 버저닝
  - MIME 기반 버저닝
- 원칙
  - 기존 필드 이름 함부로 변경/삭제 금지
  - 기본값/선택값으로 하위 호환 고려
  - 롤백 시 상위 호환도 고려
- 시맨틱 버저닝
  - major: 호환성 깨짐
  - minor: 하위 호환 기능 추가
  - patch: 하위 호환 버그 수정

## 6. 서비스 커뮤니케이션 / 게이트웨이 / 이그레스 / 서비스 메시

### 트래픽 방향
- **East-West**: 클러스터 내부 서비스 간
- **North-South**: 외부 클라이언트 <-> 내부 서비스

### 통신 방식
- 동기식 요청/응답
- 비동기식 큐/이벤트
- pub/sub

### 동기식 문제
- 쓰레드 대기
- 전체 지연 증가
- 연쇄 장애

### 비동기식 장점
- 느슨한 결합
- 최종적 일관성 확보 쉬움
- 장애 전파 감소

### 프로토콜 키워드
- HTTP
- WebSocket
- HTTP/2
- gRPC
- MQTT
- AMQP
- JSON vs Protocol Buffers

### 멱등성
- 같은 요청 여러 번 처리해도 결과 동일
- 재시도/중복 메시지 환경에서 중요
- 고유 ID로 중복 제거

### 게이트웨이
- 클라이언트 요청의 진입점
- 기능
  - SSL 오프로딩
  - 인증/권한
  - 라우팅
  - 정적 자원 전달
  - rate limit
  - retry
  - circuit breaker
  - 캐시/압축/로깅
- 구현 예
  - Nginx, HAProxy, Envoy

### Ingress / Egress
- **Ingress**: 외부 -> 내부 서비스 라우팅
- **Egress**: 내부 -> 외부 서비스 접근 제어

### 서비스 메시
- 서비스 간 공통 통신 기능 분리
- 프록시 계층으로 트래픽/보안/장애처리/관측성 담당
- 기능
  - 트래픽 관리
  - 재시도
  - 타임아웃
  - circuit breaker
  - mTLS
  - tracing
  - metrics
- 대표 도구 키워드
  - Istio
  - Linkerd
  - Consul Connect

## 7. 데이터 다루기 핵심

### 스토리지 유형
- **오브젝트 스토리지**: 대용량 파일, 저렴, 확장성 큼
- **파일 스토리지**: 공유 파일 시스템
- **블록 스토리지**: DB/OS 디스크, 빠른 I/O
- **아카이브 스토리지**: 장기 보관, 매우 저렴, 느림
- **인메모리 스토리지**: 초고속, 휘발성, 고비용

### 데이터베이스 유형
- 키/값
- 문서형
- 관계형
- 그래프
- 컬럼 패밀리
- 시계열
- 검색 인덱스

### 선택 기준
- 기능 요구
  - 데이터 형식
  - 읽기/쓰기 패턴
  - 관계 구조
  - 일관성 모델
  - 동시성
  - 검색/인덱싱
- 비기능 요구
  - 성능
  - 확장성
  - 가용성
  - 백업/복구
  - 비용
  - 팀 경험

### CDC / ETL / 데이터 레이크
- **CDC**: 변경 사항 추적
  - 알림
  - 캐시 무효화
  - 감사 로그
  - 검색 인덱스 갱신
  - 분석
- **ETL**: 추출 -> 변환 -> 적재
- **데이터 레이크**: 분석/리포팅용 통합 저장
- 주의
  - 서비스 DB를 분석팀이 직접 읽으면 결합도 증가

### 클라이언트 데이터 접근
- 제한된 토큰(발렛 키)
- GraphQL

### 확장 키워드
- **샤딩**: 수평 분할
- **캐싱**: TTL, CDC 기반 무효화, 백그라운드 갱신
- **CDN**: 사용자 가까운 위치에서 콘텐츠 제공, 지연 감소, 가용성 향상

## 8. 손필기 핵심 보강 파트

> 아래는 교안 내용을 손필기식으로 다시 붙인 **암기용 문장**입니다.  
> 시험 직전에는 이 파트만 빠르게 읽어도 흐름이 잡히게 구성했습니다.

### 8-1. 클라우드 / 클라우드 네이티브
- 클라우드 컴퓨팅 = **리소스를 빌려 쓰는 관점**
- 클라우드 네이티브 = **앱을 클라우드 체질로 만드는 관점**
- 기존 앱을 클라우드에 옮기는 것과, 처음부터 클라우드에 맞게 설계하는 것은 다름
- 클라우드 네이티브의 핵심은 **유연성 + 확장성 + 복원력 극대화**
- 빠른 배포, 자동화, 운영 간소화가 반복 키워드

### 8-2. 분산 시스템 / 12요소 앱
- 분산 시스템은 편해 보여도 **네트워크 실패를 항상 가정해야 함**
- 네트워크는 안정적이지도, 공짜도, 무한대도 아님
- 12요소 앱은 결국
  - **코드 / 설정 분리**
  - **상태 없는 프로세스**
  - **빌드 / 릴리즈 / 실행 분리**
  - **로그는 밖으로**
  - **관리 작업은 일회성으로**
- 클라우드 네이티브에서 상태를 앱 안에 두면 장애 시 곤란해짐

### 8-3. 쿠버네티스
- **도커만으로도 실행은 된다. 문제는 운영이다.**
- 쿠버네티스가 필요한 이유
  - 죽으면 다시 띄워야 함
  - 트래픽 늘면 개수를 늘려야 함
  - 서버가 여러 대면 어디서 돌지 관리해야 함
  - 무중단 배포를 하고 싶음
- 그래서
  - **도커 = 포장**
  - **PM2 = 실행**
  - **쿠버네티스 = 운영**
- 파드는 컨테이너 그 자체가 아니라 **컨테이너를 담는 최소 실행 단위**
- 서비스는 파드 IP가 자주 바뀌니까 **고정된 접근 지점** 역할
- 핵심 사고
  - 내가 직접 복구하는 게 아니라
  - **원하는 상태를 선언하면 시스템이 맞춰준다**
- desired state / 선언형 / 자동 복구 / 자동 확장 이 네 단어를 같이 묶기

### 8-4. 컨테이너 / 서버리스 / 함수
- 컨테이너 = **앱 + 실행 환경을 한 덩어리로 묶은 배포 단위**
- VM보다 가볍고 시작이 빠르며 scale-out에 유리
- 서버리스 = 서버가 없는 게 아니라 **서버 관리를 내가 안 하는 것**
- 함수(FaaS)는
  - 짧고
  - 단순하고
  - 독립적인 작업에 적합
- 오래 걸리는 작업, 특수 하드웨어, 상태 보존이 필요하면 함수보다 서비스 쪽이 적합

### 8-5. 마이크로서비스
- 마이크로서비스 = **작고 독립적인 서비스들의 조합**
- 핵심은 **느슨한 결합 + 높은 응집도**
- 서비스별로 개발, 배포, 확장이 따로 가능
- 장점만 보면 좋아 보이지만
  - 네트워크 통신 증가
  - 데이터 일관성 문제
  - 모니터링/로깅 복잡성
  - 버전 호환성 관리
  - 장애 대응
  가 같이 따라온다

### 8-6. 앱 설계
- 클라우드 네이티브는 **장애가 없기를 기대하는 구조가 아니라, 장애가 난다고 가정하는 구조**
- 전통 구조는 상태를 앱 인스턴스 안에 두기 쉬움
- 클라우드 네이티브는 **상태를 외부 저장소로 빼서 인스턴스를 가볍게 유지**
- 운영 효율성은
  - 자동화
  - 모니터링
  - 문서화
  - 점진적 변경
  - 장애 테스트
- 보안은 한 군데만 막는 것이 아니라 **심층 방어**
- reliability와 availability는 비슷해 보여도 다름
  - reliability = 장애가 있어도 제대로 동작하느냐
  - availability = 일정 시간 동안 쓸 수 있느냐

### 8-7. 서비스 통신 / 게이트웨이 / 메시
- 동기식은 간단하지만 기다림이 생기고 연쇄 장애가 쉬움
- 비동기식은 느슨한 결합에 유리하고 장애 전파를 줄이기 쉬움
- 요청/응답, 큐, pub/sub를 상황별로 선택
- 멱등성은 **재시도해도 결과가 꼬이지 않게 만드는 성질**
- 게이트웨이는 단순 문지기가 아니라
  - SSL
  - 인증/권한
  - 라우팅
  - 캐시
  - 재시도
  - 서킷 브레이커
  같은 공통 기능을 앞단에서 처리
- Ingress = 외부에서 안으로 들어오는 길
- Egress = 내부에서 외부로 나가는 길
- 서비스 메시는 통신 코드를 서비스 안에 흩뿌리지 않고 **프록시 계층으로 분리**하는 느낌

### 8-8. 데이터
- 저장소 선택은 무조건 하나가 최고가 아니라
  - 속도
  - 확장성
  - 비용
  - 접근 방식
  의 균형 문제
- 오브젝트 = 싸고 크고 오래
- 블록 = 빠르고 DB용
- 인메모리 = 매우 빠르지만 비쌈
- CDC는 데이터 변경을 잡아서
  - 캐시 무효화
  - 감사
  - 검색 반영
  - 분석
  등에 쓰는 핵심 도구
- ETL은 분석용 데이터 흐름
- 데이터 레이크는 분석/리포팅용 큰 저장소
- 서비스 DB를 분석팀이 직접 읽게 하면 결합도가 올라감
- 샤딩 = 데이터를 가로로 나눠서 확장
- 캐싱은 빠르지만 **원본과 동기화 문제**를 항상 같이 봐야 함
