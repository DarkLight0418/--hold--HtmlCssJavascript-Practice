“피그마가 원래 불편하게 만든다”기보다, **재사용성과 일관성을 지키기 위해 수정 자유도를 의도적으로 구분해 둔 구조**라고 보는 편이 더 정확합니다. Figma의 컴포넌트는 **메인 컴포넌트**와 **인스턴스**로 나뉘고, 인스턴스는 메인과 연결된 상태에서 일부 속성만 바꾸게 설계되어 있습니다. 그래서 텍스트·색·크기 같은 일부 속성은 바꿀 수 있어도, **레이어 추가/삭제·순서 변경·구조 변경**은 막혀 있는 경우가 많습니다. ([Figma Help Center][1])

## 1) 왜 “조정 자유도가 떨어진다”는 느낌이 드는가

피그마에서 답답함을 만드는 핵심은 보통 세 가지입니다.

첫째, **지금 잡고 있는 대상이 인스턴스인지**입니다. 인스턴스는 메인 컴포넌트의 규칙을 따르기 때문에, 아무거나 직접 뜯어고칠 수는 없습니다. 대신 Figma는 **component properties**를 통해 “어떤 부분은 바꿔도 된다”를 노출할 수 있게 해 둡니다. 예를 들어 텍스트, 아이콘 교체, 표시/숨김 같은 부분입니다. ([Figma Help Center][2])

둘째, **Auto Layout**입니다. 오토 레이아웃은 내용 변화와 화면 크기에 맞춰 레이아웃이 자동으로 반응하게 만드는 기능이라, 손으로 막 끌어다 놓는 방식과는 충돌할 수 있습니다. 그래서 “왜 내가 옮기면 다시 정렬되지?” 같은 느낌이 생깁니다. 최근 UI3 기준으로도 오토 레이아웃은 오른쪽 패널의 **Layout/Auto layout** 영역에서 조절합니다. ([Figma Help Center][3])

셋째, **Constraints / Resizing 규칙**입니다. 부모 프레임이 커지거나 줄어들 때 자식 레이어가 어떻게 따라갈지 정하는 규칙인데, 이걸 이해하지 못하면 “분명히 움직였는데 크기 바뀌니까 다 망가진다”가 반복됩니다. 특히 공식 문서상 **constraints는 frame 안의 layer에만 적용**되고, **auto layout frame 안의 layer에는 일반 constraints를 적용할 수 없습니다.** 이 차이를 모르면 계속 헷갈립니다. ([Figma Help Center][4])

## 2) 실무적으로 어떻게 다루면 덜 어렵나

### 1단계: 먼저 “이게 뭐인지”부터 판별

선택했을 때 오른쪽 패널에서 컴포넌트 이름이 보이거나, component properties가 보이면 **인스턴스**일 가능성이 큽니다. 이 경우 “왜 수정이 안 되지?”가 아니라,
**A. 인스턴스에서 허용된 수정만 할 것인지**
**B. 메인 컴포넌트로 가서 구조를 바꿀 것인지**
를 먼저 결정해야 합니다. UI3에서는 속성 패널에서 **Go to main component**로 바로 이동할 수 있습니다. ([Figma Help Center][3])

### 2단계: 위치/간격 문제가 나면 Auto Layout부터 의심

드래그가 마음대로 안 되면 대개 오토 레이아웃이 잡고 있는 상태입니다. 이때는 손으로 억지로 옮기기보다, **방향(가로/세로), spacing, padding, hug/fill/fixed** 같은 레이아웃 규칙을 먼저 보는 게 맞습니다. 오토 레이아웃을 해제하면 일반 프레임 속성으로 돌아갈 수 있지만, **컴포넌트 인스턴스에서는 오토 레이아웃을 직접 제거할 수 없고**, 메인 컴포넌트를 수정하거나 detach가 필요합니다. ([Figma Help Center][5])

### 3단계: 오토 레이아웃 안에서도 “예외 배치”는 가능

모든 것을 완전히 규칙대로만 두어야 하는 건 아닙니다. UI3에서는 예전 absolute position 개념이 **Ignore auto layout**로 바뀌었고, 이를 사용하면 오토 레이아웃 프레임 안에서도 특정 자식 요소를 규칙에서 빼서 수동 배치할 수 있습니다. 배지, 떠 있는 아이콘, 구석 장식 요소 같은 데 유용합니다. ([Figma Help Center][3])

### 4단계: Detach는 마지막 수단

정말 자유롭게 편집하고 싶다면 인스턴스를 **Detach**할 수 있습니다. 다만 detach하면 메인 컴포넌트와의 연결이 끊기고, 공식 문서 기준으로 **나중에 다시 그 연결을 복구할 수 없습니다.** 그래서 학습 초반에는 “안 되면 detach” 습관보다, 먼저 **메인/인스턴스 구조를 이해하는 방향**이 더 중요합니다. ([Figma Help Center][1])

## 3) 아주 현실적인 예시

예를 들어 버튼 컴포넌트를 만들었다고 해보겠습니다.

- 버튼 글자만 바꾸고 싶다 → 인스턴스에서 텍스트 override 또는 component property로 처리
- 버튼 안 아이콘을 다른 아이콘으로 바꾸고 싶다 → swap 가능한 property로 열어두면 인스턴스에서 교체 가능
- 버튼 안에 새 레이어를 추가하고 구조를 바꾸고 싶다 → 메인 컴포넌트 수정이 맞음
- 버튼 옆에 작은 뱃지를 자유롭게 띄우고 싶다 → 오토 레이아웃 내부에서 **Ignore auto layout**을 검토
- “그냥 다 내 마음대로 바꾸고 싶다” → detach 가능하지만 재사용성은 잃음 ([Figma Help Center][2])

즉, 피그마는 그림판처럼 “한 번 만든 걸 매번 개별적으로 손보는 도구”라기보다, **반복 사용될 UI를 규칙 있게 운영하는 도구**에 더 가깝습니다. 그래서 초반엔 자유도가 낮아 보이지만, 익숙해지면 오히려 수정이 빨라집니다. ([Figma Help Center][6])

## 6) 요약

정리하면,
피그마가 원래 조정 자유도를 일부 제한하는 건 맞지만, 그 목적은 **불편하게 하려는 것**이 아니라 **일관성과 재사용성을 확보하려는 것**입니다.
초보자가 가장 먼저 구분해야 할 것은 딱 세 가지입니다.

- 지금 선택한 게 **인스턴스인가**
- 지금 움직임을 잡고 있는 게 **오토 레이아웃인가**
- 지금 반응형 동작을 결정하는 게 **constraints/resizing인가**

이 세 가지만 구분되면 체감 난도가 크게 내려갑니다. ([Figma Help Center][1])

생각을 더 넓혀보면, 피그마를 배울 때의 핵심 질문은 이것입니다.
**“나는 지금 일회성 화면을 그리는 중인가, 아니면 반복 사용될 UI 시스템을 만드는 중인가?”**
이 질문에 답이 생기면, 언제 자유롭게 그려야 하고 언제 컴포넌트 규칙을 세워야 하는지가 훨씬 선명해집니다.

지금 겪는 답답한 화면을 캡처로 보여주시면, 제가 **인스턴스 문제인지 / 오토 레이아웃 문제인지 / constraints 문제인지** 바로 분류해서 설명드리겠습니다.
